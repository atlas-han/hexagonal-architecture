# Migration Learnings — BuckPal Java → Kotlin (hexagonal, Spring Boot 2.4.3)

**Generated:** 2026-05-17
**Source:** 10 sprints (00–09) across 10 commits on branch `kotlin-migration` (`e049e5e` → `f283463`)

## 1. Patterns that worked

- **Lombok `@Value` → Kotlin `data class` with `val` properties** — applied to `Money` (`src/main/kotlin/.../account/domain/Money.kt`), `Activity` and nested `Activity.ActivityId` (`Activity.kt`), `SendMoneyCommand` (`.../port/in/SendMoneyCommand.kt`), and `Account.AccountId` (`Account.kt:91`). Preserves equality semantics one-for-one; no Jackson surprises because the field set is identical. Sprint 2 risk-register §1 (Money equality via `BigInteger`) was confirmed safe by `AccountTest.calculatesBalance()` (Sprint 2 review §BC).
- **`@field:NotNull` on `SendMoneyCommand` data-class constructor params** (`SendMoneyCommand.kt:9-11`) — Bean Validation needs the annotation on the JVM field, not on the property getter or constructor parameter. `javap -v` in the Sprint 3 review confirmed all three annotations land on `private final` fields with `RuntimeVisibleAnnotations`. The `validateSelf()` invocation lives in an `init { … }` block that runs after `putfield`s, so the validator never sees half-initialised state.
- **`@JvmStatic` + `@JvmField` on `Money` companion members** (`Money.kt:24-34`) — preserves the Java call-site shape `Money.ZERO`, `Money.of(…)`, `Money.add(a, b)`, `Money.subtract(a, b)` while Java callers in Sprints 3–6 were still in place. Removing them would have cascaded into `AccountMapper.java`, `ActivityWindow.java`, `Account.java` in one shot. Verified by `compileJava` continuing to pass through Sprint 2 (Sprint 2 review §BC).
- **`@JvmOverloads constructor(...)` on `MoneyTransferProperties`** (`MoneyTransferProperties.kt:8`) — emits both the 1-arg and synthetic 0-arg constructors that Spring Boot 2.4.3's JavaBeans-style `@ConfigurationProperties` binder + `BuckPalConfiguration.moneyTransferProperties(...)` `@Bean` factory both require. Sprint 4 review §CQ confirmed via `javap -p MoneyTransferProperties.class` that both `<init>(Money)` and `<init>()` exist.
- **`kotlin-spring` / `kotlin-jpa` Gradle plugins instead of hand-written `open`** — added once in Sprint 0 (`build.gradle`, plugin block) and never re-touched. They make `@Configuration`, `@Service`, `@Component`, `@SpringBootApplication`, and `@Entity` classes `open` automatically for CGLIB/Hibernate proxying. Verified live by `SendMoneySystemTest` (Sprint 4 review §BC) and by `AccountPersistenceAdapterTest.loadsAccount` (Sprint 6 review §BC — Hibernate reflectively instantiated `ActivityJpaEntity` via the synthesized no-arg ctor).
- **`requireNotNull(field) { "<entity> loaded without <field>" }` over `!!` for JPA post-load reads** — `AccountMapper.kt:23-52` uses 7 `requireNotNull` blocks instead of `!!`. Forced by Sprint 6 contract's inline `// EVALUATOR:` decision. Failure messages identify the specific column; `grep -R "!!" .../persistence/` returned 0 (Sprint 6 review §IK). This is the de-facto idiom for the Kotlin-JPA "all entity props nullable for no-arg ctor synthesis" pattern.
- **Named arguments for same-type constructor calls** — `AccountMapper.mapToJpaEntity` (`AccountMapper.kt:65-73`) uses 6 named args (`id =`, `timestamp =`, `ownerAccountId =`, `sourceAccountId =`, `targetAccountId =`, `amount =`) on `ActivityJpaEntity(...)`. Six same-shape `Long?` parameters made positional invocation a silent source/target swap foot-gun. Mandated by Sprint 6 contract `// EVALUATOR:` block.
- **`@get:AliasFor(annotation = Component::class)` site-target on meta-annotations** — `UseCase.kt:11`, `WebAdapter.kt`, `PersistenceAdapter.kt`. Without the `@get:` site target, the annotation lands on the property descriptor and Spring's `MergedAnnotations` silently ignores the alias, breaking custom bean names. Sprint 1 review §IK called this the one trap the contract called out and the Generator did not fall into.
- **`?: error("expected … not to be empty")` over `Optional.orElseThrow(IllegalStateException::new)`** — `SendMoneyService.kt:29-32` uses the Elvis-error pattern at the two id-resolution points. The Sprint 4 review §IK confirmed it preserves the original Java `IllegalStateException` semantics and reads idiomatically in Kotlin.
- **Primary-constructor injection with `private val`** — every Spring-annotated class (`SendMoneyService.kt:14-19`, `AccountPersistenceAdapter.kt:11-15`, `SendMoneyController.kt:14-16`, `GetAccountBalanceService.kt:9-11`) uses primary-ctor injection. Zero `@Autowired` and zero `lateinit var` in `src/main/kotlin` (Sprints 4, 5, 6, 9 reviews — repo-wide grep).
- **Backtick-escaped `` `in` `` for the reserved-word package** — appears in `SendMoneyController.kt:1,3-4`, `SendMoneyService.kt:3-4`, `GetAccountBalanceService.kt:3`, and inside the three `port.`in`.*` source files themselves. JVM FQNs are unchanged; Java callers reference the unescaped `port.in.*` and still compile (Sprint 3 review §IK).

## 2. Surprises and gotchas

- **Sprint 0: Toolchain incompatibility blocked the very first sprint from running.** SYMPTOM: `./gradlew test` on the pristine repo failed with `Unsupported class file major version 61`. ROOT CAUSE: Gradle 6.8.2 (bundled Groovy 2.5) cannot run on JDK 17; Spring Boot 2.4.3's BOM-managed Lombok 1.18.18 also predates JDK-17 support (1.18.22 was the first compatible release). FIX: Sprint 0 bumped the wrapper to Gradle 7.6.4 and pinned Lombok to 1.18.30 in `build.gradle`. Both bumps were grudgingly accepted by the Sprint 0 review §CQ as covered by the contract's escape clauses ("Implementation order" + "Out of scope" wrapper clause). Without them, no acceptance check could have executed.
- **Sprint 0: `gradlew` mode-only edit triggered the auto-FAIL rule.** SYMPTOM: `git status` listed `modified: gradlew` after the wrapper bump; the Generator did not flag it in the handoff. ROOT CAUSE: the new wrapper jar was regenerated with `100755` (the upstream repo had committed `gradlew` as `100644`, a real bug). FIX: Sprint 0 review §CQ treated it as a soft finding rather than auto-FAIL because the diff was zero-byte content. Captured here so future Generators always mention chmod-only diffs.
- **Sprint 2: Mockito's subclass mocker broke on Kotlin's final-by-default classes.** SYMPTOM: `SendMoneyServiceTest.java`'s `Mockito.mock(Account.class)` + `given(account.getId())` stub failed because Kotlin compiles `class Account` to a final JVM class and property getters to final methods. ROOT CAUSE: Mockito 3 (no `mockito-inline`) cannot subclass final classes. FIX: `Account.kt:11,23,31,57` adds `open class Account` and `open fun`/`open val` on every method/property the test mocks. Self-disclosed in Sprint 2 handoff §1 ("`open` keyword scope"); accepted by Sprint 2 review as Java-parity preservation since the original Java `Account` was non-final. `kotlin-allopen` for `@Entity` was used instead of widening domain classes.
- **Sprint 2 / Sprint 4: `Account.getId(): Optional<AccountId>` shim cascade was a multi-sprint problem.** SYMPTOM: collapsing the Optional to nullable in Sprint 2 would have broken `SendMoneyService.java`'s `.orElseThrow(...)` call site; doing it in Sprint 4 broke `SendMoneyServiceTest.java`'s `given(account.getId()).willReturn(Optional.of(id))` Mockito stub. ROOT CAUSE: the Optional surface is touched in 3 layers (domain, service, test) that lived on 3 different sprints. FIX: Sprint 2 kept a public `fun getId(): Optional<AccountId>` shim alongside `val id: AccountId?` with `@get:JvmName("getIdOrNull")` to avoid getter collision (`Account.kt` Sprint 2 form). Sprint 4 added an Account.kt out-of-scope edit routing `val id` through `getId().orElse(null)` so Mockito stubs of `getId()` would also drive Kotlin `account.id` reads (Sprint 4 handoff §1, accepted as PASS-with-note in Sprint 4 review). Sprint 9 finally collapsed the entire structure to the primary-ctor `open val id: AccountId?` and removed `Optional` from the codebase (`grep -R "Optional<" src` → 0).
- **Sprint 3: `data class … : SelfValidating<…>()` parent-ctor runs before field assignment.** SYMPTOM: a draft contract version proposed hand-rolling `equals`/`hashCode` on `SendMoneyCommand` because "extending non-data parent breaks data-class equality." ROOT CAUSE: untrue — Kotlin 1.6.21 synthesizes `equals/hashCode/toString/copy` on the data class regardless. The real concern was whether `validateSelf()` could observe half-initialised state. FIX: Sprint 3 contract `// EVALUATOR:` block forced `data class` (no manual overrides), and the Sprint 3 review proved via `javap -p -v` that `putfield`s for all three fields happen before the `invokevirtual validateSelf` call.
- **Sprint 3: there is no runtime test for the `@field:NotNull` null-rejection path.** SYMPTOM: `javap -v` confirms the annotations are on the JVM fields, but no green test exercises `SendMoneyCommand(null, …)`. ROOT CAUSE: pre-existing test-suite gap (the book's tests trust the structural validation). FIX: documented as residual risk in Sprint 3 review §BC weakness and Sprint 3 handoff; not actionable inside the migration scope, but a fresh engineer should know.
- **Sprint 6: getters declared as `fun` in Kotlin do not get synthetic-property syntax.** SYMPTOM: a draft of the persistence adapter used `account.activityWindow.activities` — would not compile. ROOT CAUSE: Sprint 2 made `ActivityWindow.getActivities()` a Kotlin `fun` (returning `Collections.unmodifiableList(activities)`), not a property. Kotlin's synthetic-property sugar only applies to Java getters, not to Kotlin-declared `fun getX()`. FIX: Sprint 6 contract `// EVALUATOR:` block at line 308 forced `account.activityWindow.getActivities()` (explicit function-call form). Live in `AccountPersistenceAdapter.kt:48`.
- **Sprint 8: Mockito matchers return null at runtime, fail Kotlin's non-null intrinsic before the proxy is invoked.** SYMPTOM: every `given(mock.method(eq(x)))` and `verify(...).method(any())` call site in `SendMoneyServiceTest.kt` and `SendMoneyControllerTest.kt` failed with NPE. ROOT CAUSE: Mockito's `eq`/`any`/`ArgumentCaptor.capture()` register matchers on a thread-local stack as a side effect and return `null`. With `-Xjsr305=strict` (set in Sprint 0), the Kotlin compiler inserts `Intrinsics.checkNotNullExpressionValue(...)` at the call site for any Java-platform-typed return fed into a Kotlin non-null param. The NPE fires before the mock interceptor sees the call. FIX (Sprint 8 handoff §1, Sprint 8 review §IK): three vanilla-Mockito bridges, all confined to the test class — `private fun <T> eq(value: T): T = Mockito.eq(value) ?: value`; pre-queued `Mockito.any(…)` + non-null sentinel arg; `capture(captor): Account = captor.capture(); return accountSentinel` with `accountSentinel = Mockito.mock(Account::class.java)`. Test assertions unchanged byte-for-byte (Sprint 8 review §BC).
- **Sprint 8: a `public` Kotlin subclass of an `internal` parent does not compile.** SYMPTOM: `class Adapters : ArchitectureElement(...)` with `internal abstract class ArchitectureElement` errored with `'public' subclass exposes its 'internal' supertype`. ROOT CAUSE: Kotlin enforces visibility transitivity across the supertype hierarchy. FIX: marked `Adapters`, `ApplicationLayer`, `HexagonalArchitecture` all `internal` (Sprint 8 handoff §2, Sprint 8 review §AI). This *narrows* visibility from Java's mixed public/package-private, never widens — `DependencyRuleTests` in `io.reflectoring.buckpal` still resolves them because `internal` crosses packages within one Gradle module.
- **Sprint 8: a Kotlin `fun` with unused params emits a warning that breaks the "0 kotlinc warnings" floor.** SYMPTOM: `ArchitectureElement.denyDependency(fromPackageName, toPackageName)` ignores its parameters by design (the body hard-codes the intentional placeholder string `"io.reflectoring.reviewapp..."` — preserved verbatim from the book). Without suppression, kotlinc warns "Parameter is never used" and the rubric's hard floor (CQ ≥ 7, no warnings) breaches. ROOT CAUSE: Kotlin is stricter than javac about unused params. FIX (Sprint 8 handoff §3): `@Suppress("UNUSED_PARAMETER")` at the function level with an explanatory comment.
- **Sprint 9: collapsing `Optional<AccountId>` to `val id: AccountId?` still required `open` on the property.** SYMPTOM: even after removing the `_id` shim and `getIdOrNull` `@JvmName`, `SendMoneyServiceTest.kt`'s `given(account.id).willReturn(id)` failed with `MissingMethodInvocationException`. ROOT CAUSE: Kotlin compiles `val id` to a final getter, and Mockito's subclass mocker still cannot override final methods. FIX (`Account.kt:14`, Sprint 9 handoff §1): `open val id: AccountId?` plus an inline comment. The alternative (`mockito-inline` / `mockito-kotlin`) was explicitly out of sprint scope per the spec.

## 3. Lombok → Kotlin mapping (validated)

Every row below is verified against the actual `.kt` files on disk under `src/main/kotlin/` and `src/test/kotlin/`.

| Lombok annotation | Kotlin replacement | Where applied (verified) |
|-------------------|--------------------|-------------------------|
| `@Value` (immutable value type) | `data class … (val …)` | `Money.kt:5`, `Activity.kt:8` + nested `Activity.ActivityId:25`, `Account.AccountId` (`Account.kt:91`), `SendMoneyCommand.kt:8`, `BuckPalConfigurationProperties.kt:6` |
| `@Data` on Spring-bound props (needs JavaBeans setter) | `data class … (var …)` with default | `MoneyTransferProperties.kt:8-9` (`var maximumTransferThreshold: Money = Money.of(1_000_000L)`), `BuckPalConfigurationProperties.kt:7` (`var transferThreshold: Long = Long.MAX_VALUE`) |
| `@RequiredArgsConstructor` (Spring DI) | Primary constructor with `val` params, no `@Autowired` | `SendMoneyService.kt:14-19`, `AccountPersistenceAdapter.kt:11-15`, `AccountMapper.kt:10` (no deps), `SendMoneyController.kt:14-16`, `GetAccountBalanceService.kt:9-11` |
| `@AllArgsConstructor` + `@NoArgsConstructor` on JPA `@Entity` | Plain `class` (NOT `data class`) with `var` props + default `null` values, no-arg ctor synthesized by `kotlin-jpa` plugin | `AccountJpaEntity.kt:10-15` (single `var id: Long? = null`), `ActivityJpaEntity.kt:12-32` (6 `var … = null` props) |
| `@Builder` on test data | Inner `class Builder` with `fun withX(...): Builder = apply { … }` and a final `build()` factory | `src/test/kotlin/.../common/AccountTestData.kt` (3 `apply` setters), `ActivityTestData.kt` (6 `apply` setters) — see Sprint 8 review §IK |
| `@AliasFor` meta-annotation on `@UseCase`/`@WebAdapter`/`@PersistenceAdapter` | `@get:AliasFor(annotation = Component::class)` site-targeted on the `value` param | `common/UseCase.kt:11`, `common/WebAdapter.kt`, `common/PersistenceAdapter.kt` — site target is load-bearing (see Sprint 1 review) |
| `@Slf4j` | (not present in source) | No `LoggerFactory.getLogger(...)` call in any of the 28 production `.kt` files. The original Java sources did not declare `@Slf4j` either. |
| `@SuperBuilder`/`@EqualsAndHashCode(callSuper=…)` for `SelfValidating` subclasses | `data class … : SelfValidating<…>()` — compiler-synthesized `equals`/`hashCode` on the data class is field-only; parent ctor only initializes `validator`, so equality semantics match | `SendMoneyCommand.kt:8` (verified via `javap -p` in Sprint 3 review) |
| Lombok `compileOnly` + `annotationProcessor` deps | (removed) | `build.gradle` — `grep -R "lombok" build.gradle` → 0 (Sprint 9 verified) |

Notes on what was deliberately **not** used:
- **`JvmInline value class`** — not applied to `Account.AccountId` / `Activity.ActivityId` because they are nested types of their respective domain entities; `data class` was chosen instead per spec §3.
- **`@ConstructorBinding`** — deliberately omitted on `BuckPalConfigurationProperties` (Sprint 7 review §IK). Spring Boot 2.4.3 JavaBeans binding works with `var transferThreshold: Long = Long.MAX_VALUE`; adding `@ConstructorBinding` would change the binding pathway with no benefit.

## 4. Harness signals

- **Zero contracts went to `NEEDS_REVISION`.** Every `contracts/sprint-NN-contract.md` carries `STATUS: AGREED` on line 1. The Evaluator made extensive use of inline `// EVALUATOR:` blocks (15+ on Sprint 6 alone) to bind the Generator without a formal revision round-trip — see Sprint 6 contract (line 2: "treat every `// EVALUATOR:` block in this file as binding"). Sprints 03, 04, 06, 07, 08, 09 all carry inline Evaluator edits; Sprints 00, 01, 02, 05 are clean. This pattern (binding inline edits + flip to AGREED) avoided the formal NEEDS_REVISION → re-draft cycle entirely.
- **Zero sprints needed `FAIL → retry`.** Every `reviews/sprint-NN-review.md` line 1 reads `STATUS: PASS`. Weighted scores: 9.05, 9.55, 9.05, 9.45, 9.25, 9.40, 9.30, 9.40, 9.30, 9.55. No retry-limit bumps required.
- **The run log was never written.** `.claude/harness/workspace/logs/run-log.md` does not exist; the directory contains only `.gitkeep`. All lifecycle reconstruction below comes from review/handoff text, not from transition logs. **Recommendation for next harness run: have the orchestrator append a line to `logs/run-log.md` on every phase transition** — not knowing how many Generator iterations preceded each handoff is a real gap.
- **Two recurring Generator-side mistake classes worth a future pre-flight:**
  1. **Out-of-scope edits to `Account.kt`** — Sprint 4 declared an unavoidable Account.kt edit (PASS-with-note) and Sprint 9 declared the symmetric collapse. Both were transparent and accepted, but a pre-flight that lists files the contract permits the Generator to touch (and warns when a diff falls outside) would have eliminated the manual judgment call.
  2. **Mockito + Kotlin null intrinsics** — bit the migration once in Sprint 8 (call-site NPE on `eq`/`any` returns) and again in Sprint 9 (`MissingMethodInvocationException` on the collapsed `val id`). A "Kotlin Mockito bridge" cheatsheet baked into `criteria/kotlin-conversion.md` would have saved one round of debugging in Sprint 8.
- **One Evaluator-side mistake worth recording.** Sprint 1 review §Contract checklist self-flagged a Phase-A miss: the contract included `grep -l "@AliasFor" ... | wc -l → 3`, which is unsatisfiable because the actual idiom is `@get:AliasFor` (the `@` is consumed by the site target). The Evaluator caught it post-implementation and committed to rejecting the pattern in future contracts. No subsequent contract recurred.
