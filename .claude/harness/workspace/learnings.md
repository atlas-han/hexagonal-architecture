## 1. Patterns that worked

- **`@JvmInline value class Foo(val value: LocalDateTime)` + `companion object { fun now(): Foo }`** — used identically in `BaselineDate.kt` and `ActivityTimestamp.kt`. The factory keeps the wall-clock primitive (`LocalDateTime.now()`) confined to the inside of the VO declaration; every *caller* in `Account.kt`, `SendMoneyService.kt`, `GetAccountBalanceService.kt` now reads `BaselineDate.now()` / `ActivityTimestamp.now()` instead of leaking the raw primitive. ArchUnit `DependencyRuleTests` stays green because the VO only imports `java.time.LocalDateTime`, no application or adapter type.
- **Single-field VO → `@JvmInline value class`; multi-field VO → `data class`.** `BaselineBalanceFigures(val deposit: Money, val withdrawal: Money)` is a plain `data class` because `value class` only allows one field on the JVM. The sprint-03 contract codified this with a *negative* grep check (`grep -Eq '@JvmInline|value class BaselineBalanceFigures'` must exit 1), and that check passed cleanly — the shape rule is mechanically enforceable.
- **Helper method on the multi-field VO collapses the pair at the call site** — `BaselineBalanceFigures.toBaselineBalance(): Money = deposit - withdrawal`. The mapper body shrank from `Money.subtract(Money.of(depositBalance), Money.of(withdrawalBalance))` to `figures.toBaselineBalance()`. Re-uses the existing `operator fun Money.minus(Money): Money` so no `BigInteger` re-unwrap leaks into the adapter.
- **Boundary unwrap is the only place raw primitives still appear.** `AccountPersistenceAdapter.loadAccount` calls `activityRepository.findByOwnerSince(ownerAccountId.value, baselineDate.value)` — three call sites all use `baselineDate.value` to feed the HQL `:since`/`:until` parameters. `ActivityRepository.kt` is byte-identical to the pre-sprint-00 baseline (`git diff 5948495..HEAD -- ActivityRepository.kt` empty). The rule "wrap on the very next line in the adapter" was applied consistently and is now codified in `sprint-04-vo-convention.md`.
- **Mapper owns the JPA ↔ VO conversion in both directions, after `requireNotNull`.** `AccountMapper.mapToActivityWindow` does `val timestamp = requireNotNull(activity.timestamp) { "..." }` then `ActivityTimestamp(timestamp)` — wrapping always sees a non-null value. `mapToJpaEntity` unwraps via `activity.timestamp.value`. The "throws IllegalArgumentException when ActivityJpaEntity.timestamp is null" test still passes unchanged.
- **Named-argument construction at every two-field VO call site.** `BaselineBalanceFigures(deposit = Money.of(depositBalance), withdrawal = Money.of(withdrawalBalance))` in both `AccountPersistenceAdapter.kt:41` and the three `AccountMapperTest.kt` cases. Mechanically prevented the positional argument swap that the *old* `mapToDomainEntity(_, _, withdrawalBalance, depositBalance)` signature silently allowed.
- **`ActivityTestData.withTimestamp` keeps two overloads** — canonical `withTimestamp(ActivityTimestamp)` and convenience `withTimestamp(LocalDateTime)` that wraps. The convenience overload meant sprint-02 did not need to touch sprint-01-era call sites like `AccountPersistenceAdapterTest`'s `defaultActivity().withTimestamp(LocalDateTime.of(...))` — they kept compiling. The spec explicitly authorized this with "convenience overload accepting `LocalDateTime` is *optional* for test readability".
- **ArchUnit `DependencyRuleTests` enforced the layering automatically every sprint.** All five sprints (00..04) saw `DependencyRuleTests` reporting `tests=2 failures=0 errors=0` in `build/test-results/test/`. Because the new VOs live in `account/domain/` and only adapters/services import them, the "no domain → outward dependency" rule held without manual checking.

## 2. Surprises and gotchas

- **JDK 21 in the container vs Gradle 7.6.4 launcher (max JDK 19).** First `./gradlew test` after `git clone` failed with `Unsupported class file major version 65`. The product spec's "Out of scope" only listed `build.gradle.kts`, so the orchestrator landed a separate hot-fix commit `5948495` editing `build.gradle` (+6 lines, java toolchain pin to 17) and `gradle.properties` (+1 line, `org.gradle.java.home=/usr/lib/jvm/java-17-openjdk-amd64`) — `apt install openjdk-17-jdk-headless` first. After the pin, `./gradlew test` ran `BUILD SUCCESSFUL in 1m 33s`. Recorded in run-log entry `2026-05-27T02:36:30Z` and audited by the sprint-00 Evaluator: hot-fix touched 0 files under `src/`, was committed separately from sprint-00 (`b3ddc61`), so the "no production code changes" invariant for sprint-00 remained trivially satisfied. **Future runs on this container depend on this pin** — any unpin re-triggers the JDK-21 failure.
- **Mockk 1.13.8 cannot bind matchers to a `@JvmInline value class` parameter slot.** sprint-01's first build was red with five `NullPointerException`s inside `SignedCall.toString` (chain: `String.valueOf → AbstractCollection.toString → StringConcatHelper.stringOf → LocalDateTime.toString`). Root cause: the Kotlin compiler emits `loadAccount-6vJy7ms(AccountId, LocalDateTime)` on the JVM ABI (value class is erased to its underlying primitive), but mockk's matcher table tracks the boxed `BaselineDate`. Adding `registerInstanceFactory` for both `LocalDateTime` and `BaselineDate` cleared the NPE but then mockk emitted `MockKException: Failed matching mocking signature ... left matchers: [matcher<BaselineDate>()]` because slot-attribution uses the boxed type while the recorded arg list uses the unboxed JVM type. `any<BaselineDate>()`, `match { true }`, and explicit instance factories all hit the same dead end. The contract's *Risk #2* explicitly predicted "Mockk `any()` continues to work because `any()` is generic" — that turned out to be empirically wrong. **Fix:** replace the `LoadAccountPort` mockk mock with hand-rolled test doubles (`RecordingLoadAccountPort` in `GetAccountBalanceServiceTest`, `StubbedLoadAccountPort` in `SendMoneyServiceTest`) that implement the port directly. All other mockk mocks (`Account`, `AccountLock`, `UpdateAccountStatePort`) work fine because their parameters are not value classes. The replaced `verify(exactly = 0) { loadAccountPort.loadAccount(any(), any()) }` became `loadAccountPort.loadCallsByAccountId.size shouldBe 0` — semantically identical (Evaluator: "no weakening").
- **Sprint-02 contract check 5 was self-contradictory.** Hard exit criterion mandated `companion object { fun now(): ActivityTimestamp = ActivityTimestamp(LocalDateTime.now()) }` (literal `LocalDateTime.now()` in the body). Acceptance check 5 demanded `grep -RnE 'LocalDateTime\.now\s*\(\s*\)' src/main/kotlin/io/reflectoring/buckpal/account/domain/` return **zero matches**. Both constraints cannot hold simultaneously because the factory body is itself inside `account/domain/`. Furthermore, the same grep would scan sprint-01's `BaselineDate.kt:11` which the sprint-02 contract listed as "Unchanged" (so the Generator could not delete the line there either). The Generator flagged this transparently in handoff Note 1 rather than mutating the spec; the Evaluator ruled **PASS by literal-vs-intent**: "intent ('cutoff factory lives on the VOs') is fully met; every *caller* of `LocalDateTime.now()` in the domain folder is gone — `Account.kt`'s two `Activity(... LocalDateTime.now() ...)` call sites both became `ActivityTimestamp.now()`." Recorded as a contract-wording bug for retrospective tightening (suggested fix: scope the grep to `Account.kt` only or `--exclude` the two VO files).
- **Argument-order swap was the high-value risk in sprint-03.** Old positional call: `mapToDomainEntity(_, _, withdrawalBalance: Long, depositBalance: Long)` — `withdrawalBalance` first, `depositBalance` second. New named call: `BaselineBalanceFigures(deposit: Money, withdrawal: Money)` — `deposit` first, `withdrawal` second. A blind rewrite that preserved positional order would invert the arithmetic silently (baseline would compute as `withdrawal - deposit` instead of `deposit - withdrawal`). **Detected by:** `AccountMapperTest`'s working case `deposit = Money.of(700L), withdrawal = Money.of(200L)` asserts `result.baselineBalance shouldBe Money.of(500L)`. 700 − 200 = 500 — passes only if the named-arg pairing matches the semantic origin. Build green ⇒ swap correctly applied at the one production site (`AccountPersistenceAdapter.kt:41`) and all three test sites.
- **Sub-agent `Write` permission was consistently denied for `workspace/contracts/`, `workspace/handoffs/`, and `workspace/reviews/`.** Every sprint, the Planner/Generator/Evaluator sub-agents emitted their artifact bodies inline in the result message; the orchestrator then materialized them to disk. This shows up indirectly in `run-log.md` — see e.g. the 4 consecutive `STATUS=NEEDS_REVISION` entries on sprint-01-contract (`03:43:20Z..03:43:41Z`) before the eventual `AGREED` at `05:42:03Z`. Pattern stabilized by sprint-02 onward, but it puts non-trivial write-throughput pressure on the orchestrator and means harness-internal files (`logs/run-log.md`) are the only files a sub-agent can append to autonomously.
- **`ActivityWindow`'s `minByOrNull` / `maxByOrNull` could not use `Activity::timestamp` after sprint-02.** Before: `activities.minByOrNull(Activity::timestamp)` — selector returned `LocalDateTime`, sorted by its natural `Comparable<LocalDateTime>` order. After: same code would return `ActivityTimestamp`, which deliberately does **not** implement `Comparable`. **Fix:** rewrite as `activities.minByOrNull { it.timestamp.value }` — extract the wrapped `LocalDateTime`, sort by its natural order, the resulting `Activity`'s `.timestamp` is already `ActivityTimestamp` for the return. The spec called this out under "Target Kotlin conventions" ("Operator overloads only when they read naturally") and the Evaluator's idiomatic-Kotlin review (9/10) flagged the trailing-lambda selector as the right idiom.
- **`Account.baselineBalance: Money` re-typing was rejected even though it's a single-field "Money-with-meaning" case.** Looks like a textbook VO candidate (just add `value class BaselineBalance(val value: Money)` and you're done) but introducing it forces constant `.value`/wrapping ceremony whenever `baselineBalance` is added to or subtracted from any other `Money` — every `withdraw`/`deposit` arithmetic path. Rejected in sprint-00 ADR row 4 with the rule "already-wrapped primitives do not get a second wrapper", and that rule was generalized in `sprint-04-vo-convention.md` so future contributors do not re-propose it.
- **`BuckPalConfigurationProperties.transferThreshold: Long` cannot be a VO without writing a Spring `Converter`.** Spring `@ConfigurationProperties` binds primitives directly from YAML/properties. A `value class TransferThreshold(val value: Long)` would force a `org.springframework.core.convert.converter.Converter<String, TransferThreshold>` registered in a `@Configuration`. The spec correctly rejected this in sprint-00 ADR row 5; the value is already lifted into `Money.of(...)` exactly once in `BuckPalConfiguration#moneyTransferProperties`, which is the right boundary. **Verified at sprint-04:** `BuckPalApplicationTests` boots the Spring context successfully — the `transferThreshold: Long` binding still works.

## 3. Lombok → Kotlin mapping (validated)

**Note on scope.** This migration extracted VOs from an *already-Kotlin*
codebase (the Java → Kotlin migration completed in a previous run, archived
to `workspace/archive/2026-05-27-kotest-mockk`). No `import lombok` exists
anywhere in `src/` — verified by `grep -R "import lombok" src/` returning
exit 1 on every sprint. The table below therefore documents the
**primitive-leak → VO** mapping that this migration actually performed,
which is the analog of the Lombok→Kotlin table for a VO-extraction stream.

| Primitive leak | VO replacement | Where applied |
|----------------|----------------|---------------|
| `LocalDateTime` (cutoff for activity window) | `@JvmInline value class BaselineDate(val value: LocalDateTime)` + `companion object { fun now() }` + `fun minusDays(days: Long): BaselineDate` | `LoadAccountPort.loadAccount`, `SendMoneyService` (`BaselineDate.now().minusDays(10)`), `GetAccountBalanceService` (`BaselineDate.now()`), `AccountPersistenceAdapter.loadAccount`; unwrapped via `.value` at 3 `ActivityRepository.*` calls |
| `LocalDateTime` (instant when an `Activity` occurred) | `@JvmInline value class ActivityTimestamp(val value: LocalDateTime)` + `companion object { fun now() }` | `Activity.timestamp` (primary + secondary ctors), `ActivityWindow.getStartTimestamp()/getEndTimestamp()`, `Account.deposit`/`withdraw` (`ActivityTimestamp.now()`), `AccountMapper.mapToActivityWindow` (wraps), `AccountMapper.mapToJpaEntity` (unwraps via `.value`) |
| two positional `Long`s (deposit, withdrawal aggregates) | `data class BaselineBalanceFigures(val deposit: Money, val withdrawal: Money)` + `fun toBaselineBalance(): Money = deposit - withdrawal` | `AccountMapper.mapToDomainEntity(figures: BaselineBalanceFigures)` (signature shrunk from 4 params to 3), `AccountPersistenceAdapter.loadAccount` (constructs once with named args + `?: 0L` defaults) |

**Shape rules (mechanically enforced):**

| Rule | Check (contract grep) | Enforced by |
|------|------------------------|-------------|
| Single-field VO is a `@JvmInline value class` | `grep -Eq '@JvmInline' && grep -Eq 'value class Foo' && grep -Eq 'val value: <Primitive>'` | sprint-01, sprint-02 contracts |
| Multi-field VO is a `data class`, NOT `@JvmInline` | `grep -Eq '@JvmInline\|value class BaselineBalanceFigures'` must exit 1 | sprint-03 contract |
| One VO per file under `account/domain/` | `test -f .../Foo.kt` per VO | every sprint contract |
| `companion object { fun now(): Foo }` factory where the VO wraps a clock value | `grep -q 'companion object' && grep -Eq 'fun now\s*\(\s*\)\s*:\s*Foo'` | sprint-01, sprint-02 contracts |

**Mockk interop posture (validated):**

| Mockk target | Status | Notes |
|--------------|--------|-------|
| Mocking a method whose parameter is `@JvmInline value class` | **DOES NOT WORK** with mockk 1.13.8 | Replace with hand-rolled test double; recording call list + per-method assertion replaces `verify { ... }` (sprint-01) |
| Mocking a method with no value-class params (`AccountLock`, `UpdateAccountStatePort`) | Works | No changes from prior runs |
| Mocking a class with value-class fields (`Account.id: Account.AccountId?`) | Works for *method calls* on the mock; matchers `any<AccountId>()` work because `AccountId` was pre-existing and apparently exercises a different mockk codepath | sprint-01 kept `Account` as a mockk mock |

## 4. Harness signals

- **Sprint-00 contract: 1 round to `AGREED`** (`02:22:41Z` → `02:23:19Z`). Sprint-02, sprint-03, sprint-04 contracts also reached `AGREED` in 1 round each (≤90 seconds each). Sprint-01 contract was the outlier: **4 consecutive `NEEDS_REVISION` entries** (`03:43:20Z..03:43:41Z`), then a long pause (~2 hours), then `AGREED` at `05:42:03Z`. The four NEEDS_REVISION cycles were almost back-to-back (sub-20-second deltas), suggesting the Evaluator was iterating rapidly on small contract wording before the Planner/Orchestrator caught up — a candidate for tightening the contract-agreement turnaround.
- **No sprint required `FAIL → retry`.** All 5 sprint reviews recorded `STATUS=PASS` on first submission. No retry-limit bumps required. The harness loop was effectively `Planner → (Contract↔Evaluator) → Generator → Evaluator → PASS` for every sprint after sprint-00, with sprint-04 a no-op verification sprint (zero production source touched).
- **Recurring Evaluator finding across sprints: "literal grep-check vs spec intent" mismatch.** Sprint-02's contract check 5 (`LocalDateTime.now()` over `account/domain/`) and similar grep-shape checks in sprint-04 (`BaselineDate(|ActivityTimestamp(|BaselineBalanceFigures(`) both produced hits inside the VO declarations themselves. The Evaluator ruled both PASS on intent grounds, but the pattern is the same: a grep designed to forbid primitive leaks at *call sites* will always also match the *VO declaration* (where the boundary lives). **Candidate pre-flight check for the next harness run:** when a contract grep targets `account/domain/`, auto-scope it to `account/domain/Foo.kt` excluding the VO files themselves, OR phrase it as "no hit *outside* a `companion object { fun ... }` block".
- **Sub-agent Write denials drove an artifact-relay pattern.** Every sprint, sub-agents could not write directly to `workspace/{contracts,handoffs,reviews}/` and instead attached the full markdown body to their result message. The orchestrator then `Write`-tooled the file. This pattern stabilized by sprint-02 onward — no longer surfaces as failures in `run-log.md` after the planner phase. Recorded as a harness-permissions friction point, not a code defect; **candidate for tightening permissions on workspace subdirectories** so sub-agents can write the artifacts they own.
- **Gitleaks was unavailable in the container.** Every sprint review reported `GITLEAKS_VIOLATIONS: SKIPPED` because `command -v gitleaks` returned non-zero. Per evaluator playbook this is documented escape-hatch behaviour, not an automatic FAIL. Worth installing gitleaks (or pinning a Docker image that includes it) for the *next* run if security scanning is intended as a real gate.
- **ArchUnit `DependencyRuleTests` was the cheapest invariant to enforce.** It ran inside every `./gradlew test` / `./gradlew check` invocation, and every sprint's review explicitly cited `tests="2" failures="0" errors="0"` from `build/test-results/test/TEST-io.reflectoring.buckpal.DependencyRuleTests.xml`. Future migrations that need to preserve hexagonal layering should plant this rule before the first generative sprint, not after — it caught zero violations here only because the spec was scoped to keep VOs inside `account/domain/`, and ArchUnit was the proof.

## 5. Future improvements / deferred work

- **`Money.amount: BigInteger` → `Long` silent narrowing in `AccountMapper.mapToJpaEntity`** (sprint-00 ADR row 7, REJECT / OUT-OF-SCOPE). Pre-existing precision-loss risk when a `Money` exceeds `Long.MAX_VALUE`. Recorded in spec risk register. **Where to start:** redesign `Money` itself (storage type, JPA column type for `ActivityJpaEntity.amount`, all the call-site arithmetic) as a separate migration stream — not a VO extraction. Probably needs its own multi-sprint plan because `AccountJpaEntity` and `ActivityJpaEntity` columns are both `Long`-typed today and the SQL fixture `SendMoneySystemTest.sql` would need to change.
- **Spring `Converter<String, TransferThreshold>`** if you want `transferThreshold` to also be a VO (sprint-00 ADR row 5, REJECT). Today the value lifts into `Money.of(...)` exactly once at `BuckPalConfiguration#moneyTransferProperties` — the right boundary. **Where to start:** write a `org.springframework.core.convert.converter.Converter` and a `@ConfigurationPropertiesBinding` registration, then verify `BuckPalApplicationTests` still boots. Probably not worth the ceremony.
- **`SendMoneyController` `@PathVariable` re-typing** (sprint-00 ADR row 6, REJECT) is permanently out of scope — the HTTP path `POST /accounts/send/{Long}/{Long}/{Long}` is the external contract. The convention file says "do not re-litigate this".
- **`Account.baselineBalance` re-typing as a `BaselineBalance(value: Money)` wrapper** (sprint-00 ADR row 4, REJECT). Out of scope because it would force `.value`/wrap ceremony on every arithmetic call with other `Money`. Same convention rule: "already-wrapped primitives do not get a second wrapper".
- **Gradle 8 deprecation warnings** surface on every `./gradlew check` (recorded in sprint-04 review Note 5). Pre-existing. **Where to start:** a dedicated Gradle-upgrade sprint, separately from VO work. Would also let the JDK 17 pin go away once Gradle ≥8.5 is in place.
- **Mockk → mockk-jvm upgrade or migration to a value-class-aware mocking library** (sprint-01 forced hand-rolled test doubles for `LoadAccountPort`). Worth investigating whether mockk 1.13.10+ or a newer alternative fixes the value-class matcher-slot bug. If yes, restore the `verify { loadAccountPort.loadAccount(any(), any()) }` form in `SendMoneyServiceTest` / `GetAccountBalanceServiceTest`; otherwise leave the hand-rolled doubles, which are well-justified. **Where to start:** check mockk's GitHub for issues mentioning "value class" and matcher slots; reproduce against the current `BaselineDate` parameter; if fixed, bump version in `build.gradle` (the Groovy DSL, not `.kts`) and revert the two test files.
- **Contract-grep scope tightening for the next harness run** (recurring Evaluator finding §4). When a contract grep targets `account/domain/`, the Planner should auto-exclude the VO files themselves or scope to specific *caller* files. **Where to start:** add a pre-flight rule to `agents/planner.md` or `agents/evaluator.md` that flags any `grep -RnE` over `account/domain/` for inclusion of VO declarations and requires either `--exclude` or explicit-file enumeration.
